## 1. 文档概述

### 1.1 目的

定义 MVP 阶段的产品目标、用户、功能、交互、技术边界与成功指标，作为研发、设计、测试的统一依据。

### 1.2 背景

当前 AI 编程工具以线性对话为中心。项目变大后，上下文膨胀、修改定位困难、AI 迎合式生成导致错误放大。转码出身的 vibe coder 尤其痛点明显：能写代码，但难以理解架构，也不知道如何让 AI「改对地方」。

本产品将 Cursor 式开发体验升级：**中间主区域从代码视图扩展为可切换的多维可视化工作区**，实现「点哪改哪、看见 AI 在干什么、AI 代劳细节」。

产品长期愿景是囊括从**模糊 idea → 成熟产品**的完整路径；MVP 聚焦其中「开发执行」段，并以**三大可视化**（软件结构、Agent 流程、File Tree）让 vibe coding 过程 as easy as possible。

### 1.3 MVP 一句话定位

> **对标 Cursor 布局的独立 AI 开发环境：三大可视化（功能结构、Agent 流程、File Tree）让开发全程可见、可纠、可 debug，点选节点精准注入 context，AI 自动理解需求并落地代码。**

### 1.4 待确认假设（不阻塞 MVP，PRD 中先按此设计）

| 假设项 | MVP 默认 |
|--------|----------|
| 产品正式名称 | 暂用「Project OS」 |
| Prompt 精进反馈 | 默认开启，用户可「跳过直接执行」 |
| 首发支持技术栈 | TypeScript + Next.js + Node 为主 |
| 交付形态 | 桌面独立应用（Electron / Tauri），内置 Monaco 编辑器 |
| 定价 | Free + Pro $19–29/月 |

## 2. 问题陈述

### 2.1 核心问题

| 问题 | 表现 | 影响 |
|------|------|------|
| 上下文迷失 | 长对话后 AI 不知道在改什么 | 改错文件、重复劳动 |
| 修改定位难 | 用户说不清「改哪个功能」 | prompt 越来越长，效果越来越差 |
| 架构恐惧 | 转码用户看不懂技术架构 | 不敢改、全靠 AI 瞎改 |
| 需求表达模糊 | 自然语言歧义大 | AI 理解偏差，返工多 |
| 报错处理混乱 | debug 与业务需求混聊 | 效率低、挫败感强 |
| Agent 黑盒 | 不知道 AI 正在做什么、为何这样改 | 无法纠错，只能反复说「看一下哪里有问题」 |
| 打补丁式修 Bug | AI 覆盖报错而非找根因 | 问题复发、技术债累积 |
| File Tree 迷失 | 传统文件夹逐层展开，只见文件不见结构 | 找不到问题文件、不理解项目组织 |

### 2.2 MVP 要解决的问题（聚焦）

1. 让用户**看见**项目的宏观功能关系（不是技术架构）
2. 让用户**看见** Agent 正在执行的流程，并能可视化纠错
3. 让用户**看见** File Tree 的结构化全貌（树状语义视图，而非无限展开文件夹）
4. 让用户**点选**功能节点，精准注入 AI context
5. 让用户在执行前**确认 AI 是否理解需求**（Prompt 精进反馈）
6. 让 AI **自动处理**技术细节，用户只做功能级决策
7. 报错时提供**独立修复路径**（debug / business context 分离），并通过 **Harness** 确保修复根因而非打补丁

### 3.1 Primary：转码小 vibe coder（MVP）

| 属性 | 描述 |
|------|------|

## 4. 产品目标

### 4.1 业务目标（MVP 12 个月内）

- 验证「三大可视化 + 节点 context + Prompt 精进 + Harness 根因修复」闭环
- 达成 1,000+ 注册用户，100+ 付费用户
- 30 天留存 > 25%

### 4.2 用户目标

- 完成功能修改时，比纯 Cursor 聊天**减少 30% 对话轮次**
- 用户无需理解技术架构即可完成常见功能增删改
- 用户对「AI 是否理解我的需求」有清晰确认感

### 4.3 首次商业化必推能力（Paid Launch 门槛）

以下三项可视化构成首次付费上线的**核心交付**，可由**一个 Agent 统管**或**三个专项 Agent 分工**，产品层面对用户呈现为统一体验：

| 可视化层 | 用户价值 | 商业化优先级 |
|----------|----------|-------------|
| **软件结构可视化** | 看见功能关系，点选修改 | P0 — 必须有 |
| **Agent 流程可视化** | 看见 AI 在干什么，随时纠错 | P0 — 必须有 |
| **File Tree 可视化** | 看见项目文件结构全貌，快速定位 | P0 — 必须有 |

### 4.4 北极星指标

**可视化驱动编辑占比** = 通过「三大可视化中任一层级定位 → 注入 context → 发起编辑」完成的修改次数 / 总修改次数

**MVP 目标：> 40%**

## 5. 用户旅程

### 5.1 路径 A：已有项目（Brownfield）

```
打开应用 → 导入本地 repo → 自动生成功能地图（30s–2min 渐进）
→ 浏览功能关系 → 点击目标节点 → 右侧 chat 切换 context
→ 输入需求 → Prompt 精进反馈 → 确认 → AI 执行
→ Agent 流程面板实时展示执行步骤 → 预览/代码 review
→ 完成 / 报错修复（Harness 根因分析，非打补丁）
```

### 5.2 路径 B：从零开始（Greenfield）

```
打开应用 → 新建项目 → 架构对话（AI 引导敲定功能规划）
→ 生成功能地图骨架 → 逐节点开发（同路径 A 后续流程）
```

### 5.3 典型故事：给现有项目加 Google 登录

1. 用户导入 `~/my-app` repo
2. 功能图显示：`首页`、`用户中心`、`设置`；`用户认证`为灰色虚线建议节点
3. 用户点击 `用户认证` 节点（或「+ 添加功能」）
4. 右侧 chat 标题变为「Context: 用户认证」，显示当前节点摘要
5. 用户输入：「加 Google 登录，登录后跳转个人中心」
6. **Prompt 精进反馈**弹出：
   - AI 理解：新增 OAuth 登录，提供商 Google，成功后 redirect 到 `/profile`
   - 模糊点：是否已有用户数据库？是否需要绑定邮箱？
   - 建议精进：「是否保留邮箱密码登录？」
7. 用户补充确认
8. AI 自动生成并应用代码（用户不接触 Change Plan 细节）
9. 节点变实线、变绿；Preview 框可点开看登录页
10. Agent 流程面板展示：理解需求 → 分析关联文件 → 生成变更 → 应用；用户可点击某步骤暂停/纠正
11. 若终端报错 → 侧边栏「修 Bug」→ Harness 定位根因（业务节点 + 技术文件双定位）→ debug context 独立处理

### 5.4 Agent 纠错典型故事

1. 用户发起「修 Bug」后，Agent 流程面板显示：诊断 → 假设根因 → 拟定修复 → 应用
2. 用户在流程面板中发现 Agent 定位到错误文件（如误指 `utils.ts` 而非 `auth.ts`）
3. 用户直接点击流程中「分析关联文件」步骤，手动指定正确节点或文件
4. Agent 重新规划修复路径，Harness 标记前次尝试为「补丁式修复（已拒绝）」
5. 根因修复完成后，功能地图对应节点与 File Tree 对应文件同步变绿

### 6.4 F3：节点 Context 聊天（核心）

#### F3.1 Context 自动注入

| ID | 需求 | 优先级 |
|----|------|--------|
| F3.1.1 | 用户点击功能节点 → 右侧 AI chat 自动切换 context | P0 |
| F3.1.2 | Chat 标题/面包屑显示当前节点路径 | P0 |
| F3.1.3 | Context 包含：功能定义、关联文件摘要、上下游功能、最近变更 | P0 |
| F3.1.4 | 未选中节点时，chat 为项目级全局 context | P1 |
| F3.1.5 | 切换节点时，chat 历史可选择「继承」或「新开会话」 | P2 |

#### F3.2 Prompt 精进反馈（核心功能）

**功能定义**：用户输入 prompt 后、AI 执行前，Agent 自动解析意图，向用户反馈「我理解了什么 / 哪里模糊 / 如何精进」，用户确认后再执行。

| ID | 需求 | 优先级 |
|----|------|--------|
| F3.2.1 | 用户提交 prompt 后，进入「理解反馈」步骤（非立即执行） | P0 |
| F3.2.2 | 反馈卡片包含三块：**已理解** / **模糊点** / **精进建议** | P0 |
| F3.2.3 | 「已理解」：结构化复述（功能、行为、约束、影响范围） | P0 |
| F3.2.4 | 「模糊点」：列出歧义（如「登录」指 OAuth 还是邮箱？） | P0 |
| F3.2.5 | 「精进建议」：给出 1–3 条可选精进 prompt 供一键采纳 | P0 |
| F3.2.6 | 用户操作：**确认执行** / **修改 prompt** / **跳过直接执行** | P0 |
| F3.2.7 | 反馈基于当前节点 context + 项目 current work | P0 |
| F3.2.8 | 简单 prompt（如「加个保存按钮」）可快速通道，减少打断 | P1 |

**交互流程**：

```
用户输入 prompt
    ↓
Agent 解析（< 3s 目标）
    ↓
展示「理解反馈卡」
    ├── 已理解：...
    ├── 模糊点：...
    └── 建议精进：[选项A] [选项B] [我自己改]
    ↓
用户确认 → AI 执行变更
```

**设计原则**：

- 默认开启，培养用户「确认理解」习惯
- 不变成审批负担：简单需求可跳过
- 反馈语言用**用户语言**（功能级），不用技术术语

#### F3.3 AI 自动执行

| ID | 需求 | 优先级 |
|----|------|--------|
| F3.3.1 | 用户确认后，AI 自动生成代码变更 | P0 |
| F3.3.2 | 内部生成 Change Plan（对用户不可见） | P0 |
| F3.3.3 | 变更前自动创建 git 分支/检查点（可回滚） | P0 |
| F3.3.4 | 执行后展示简要结果摘要（改了哪些文件、功能状态变化） | P0 |
| F3.3.5 | 用户**不需要**手动编辑 Change Plan 或架构细节 | P0 |
| F3.3.6 | 保守策略：默认只改当前功能节点 ± 直接关联 | P0 |
| F3.3.7 | 执行全程驱动 Agent 流程面板实时更新 | P0 |

### 6.5 F4：Agent 流程可视化（首次商业化必推 ★）

**功能定义**：将 Agent 当前正在做什么以**实时流程图**呈现。用户无需反复说「看一下哪里有问题」，可直接在流程图上观察、暂停、纠正 Agent 行为。

**核心价值**：

- 消除 Agent 黑盒，建立用户信任
- 防止 Agent 以「打补丁」方式修 Bug（与 Debug Harness 联动）
- 精确定位是**哪个业务环节**或**哪个技术节点**出错

| ID | 需求 | 优先级 |
|----|------|--------|
| F4.1.1 | 主编辑区「Agent 流程」Tab，展示当前任务的实时流程图 | P0 |
| F4.1.2 | 流程节点类型：理解需求 / 检索 context / 分析文件 / 生成方案 / 应用变更 / 验证 / 修复 | P0 |
| F4.1.3 | 当前执行步骤高亮 + 进度动画（进行中 / 完成 / 失败 / 跳过） | P0 |
| F4.1.4 | 用户可点击任意已完成或进行中步骤，查看该步输入/输出摘要 | P0 |
| F4.1.5 | 用户可在流程步骤上**注入纠正**（如「跳过此文件」「重新分析 auth 模块」） | P0 |
| F4.1.6 | 修 Bug 时，流程图标注**业务节点路径**与**技术文件路径**的分叉与汇合 | P0 |
| F4.1.7 | 检测到「补丁式修复」模式时，流程节点标黄警告 | P0 |
| F4.1.8 | 历史任务流程可折叠回看（最近 10 次） | P1 |
| F4.1.9 | 流程图与功能地图、File Tree 双向联动（点击高亮对应节点/文件） | P0 |

**交互流程**：

```
用户确认 Prompt 精进
    ↓
Agent 流程面板自动展开，实时绘制流程图
    ├── Step 1: 理解需求 ✓
    ├── Step 2: 检索 context ✓
    ├── Step 3: 分析关联文件 ● (进行中，高亮)
    ├── Step 4: 生成变更方案 ○
    └── Step 5: 应用并验证 ○
    ↓
用户发现 Step 3 定位文件有误 → 点击该步骤 → 注入纠正
    ↓
Agent 从该步骤重跑，后续步骤重新生成
```

**Agent 架构选项**（实现层，对用户透明）：

| 方案 | 描述 | 适用 |
|------|------|------|
| 单 Agent | 一个 Agent 统管，流程图是其内部 step 的外显 | MVP 推荐 |
| 三 Agent | 结构 Agent / 执行 Agent / Debug Agent 分工，流程图展示协作 | 可选 |

### 6.6 F5：File Tree 可视化（首次商业化必推 ★）

**功能定义**：左侧 File Tree 不再是传统 IDE 的**无限层级文件夹展开**，而是**语义树状结构可视化**——按功能/模块/类型聚合，让用户一眼看见项目文件组织全貌。

| ID | 需求 | 优先级 |
|----|------|--------|
| F5.1.1 | 左侧 File Tree 默认展示**语义树**（按功能模块聚合），而非纯目录层级 | P0 |
| F5.1.2 | 树节点类型：功能模块 / 文件 / 配置 / 测试 / 静态资源 | P0 |
| F5.1.3 | 支持一键切换「语义树视图 ↔ 传统目录视图」 | P1 |
| F5.1.4 | 点击 File Tree 节点 → 右侧 AI chat 切换为**文件级 context** | P0 |
| F5.1.5 | 与功能地图双向联动：选中功能节点 → File Tree 高亮关联文件 | P0 |
| F5.1.6 | 与 Agent 流程联动：流程中「分析文件」步骤 ↔ File Tree 高亮 | P0 |
| F5.1.7 | 报错文件自动高亮并上浮至树顶（「需关注」区） | P0 |
| F5.1.8 | 主编辑区「File Tree」Tab 提供大图谱视图（文件关系图，可选） | P1 |

**与传统 File Tree 的差异**：

| 传统 IDE File Tree | Project OS File Tree |
|-------------------|---------------------|
| 纯目录层级，逐层展开 | 语义聚合，按功能/模块分组 |
| 只见文件夹嵌套 | 可见文件角色（路由/组件/工具/配置） |
| 与业务逻辑无关 | 与功能地图、Agent 流程联动 |
| 报错后用户自行找文件 | 报错文件自动标记并上浮 |

### 6.7 F6：Debug Harness（根因修复 ★）

**功能定义**：当检测到错误时，Harness 确保 AI **理解问题根因**并**从根因修复**，而非简单覆盖报错（打补丁）。Harness 是修 Bug 流程的质量约束层，与 Agent 流程可视化、三大可视化联动。

| ID | 需求 | 优先级 |
|----|------|--------|
| F6.1.1 | 报错触发 Harness 流程：收集证据 → 假设根因 → 验证假设 → 拟定修复 → 回归验证 | P0 |
| F6.1.2 | 修复前必须输出**根因说明**（对用户可见，非黑盒） | P0 |
| F6.1.3 | **补丁检测**：若修复仅修改报错行、未触达根因节点，Harness 拒绝执行并标黄 | P0 |
| F6.1.4 | 双定位：同时标注**业务环节**（功能地图节点）和**技术节点**（File Tree 文件/符号） | P0 |
| F6.1.5 | 修复历史记录：每次尝试标记为「根因修复 / 补丁修复（拒绝）/ 待验证」 | P0 |
| F6.1.6 | 同一报错 3 次补丁式修复后，强制升级为「根因分析模式」，切换 Agent 策略 | P1 |
| F6.1.7 | Harness 结论同步至 Agent 流程面板与功能地图节点状态 | P0 |

**Harness 流程**：

```
报错检测
    ↓
Harness 启动
    ├── 1. 收集：堆栈 + 日志 + 关联功能节点 + 关联文件
    ├── 2. 假设：生成 1–3 个根因假设（按概率排序）
    ├── 3. 验证：对每个假设提供证据/反证
    ├── 4. 修复：基于最高置信根因生成修复（非仅覆盖报错行）
    └── 5. 回归：运行验证，确认同类问题不会复发
    ↓
补丁检测器审查
    ├── 通过 → 应用修复，节点变绿
    └── 拒绝 → 流程面板标黄，提示用户「AI 试图打补丁，已拦截」
```

### 8.3 Agent 设计（MVP 单 Agent + 流程外显）

| Skill | 触发场景 |
|-------|----------|
| `decompose_features` | 项目导入 / 文件变更 |
| `understand_prompt` | 用户提交 prompt → 精进反馈 |
| `execute_change` | 用户确认后生成代码（每步驱动流程图更新） |
| `trace_root_cause` | Harness 根因分析 |
| `fix_bug` | 侧边栏修 Bug（受 Harness 约束） |
| `scaffold_project` | 新建项目架构对话 |

**流程外显**：Agent 内部每个 step 写入 `AgentFlowTracker`，UI 实时渲染为流程图。用户纠正操作写入 `FlowCorrection` 事件，Agent 从指定 step 重跑。

### 8.4 Debug Harness 架构

```
Error Event
    ↓
Harness Engine
    ├── Evidence Collector（堆栈 + 日志 + 功能节点 + 文件）
    ├── Root Cause Analyzer（LLM + 静态分析）
    ├── Patch Detector（变更行数/范围/触达节点审查）
    ├── Fix Generator（基于根因，非报错行）
    └── Regression Validator（终端/编译 re-check）
    ↓
Pass → Apply + 更新三大可视化
Reject → 流程面板标黄 + 用户可选强制/重试
```

### 8.5 Source of Truth

- **代码库** = 唯一真相
- **功能图** = 面向用户的投影层
- 用户改代码 → 图跟随更新
- 用户通过节点 chat 改功能 → AI 生成代码 → 图跟随更新
- 不一致时标黄，不阻断（MVP 不做架构纠错）

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 功能拆解不准 | 用户不信图 | 允许拖拽修正；诚实标注「推断中」 |
| Agent 流程过于复杂 | 用户看不懂 | 默认折叠为摘要模式；新手引导 |
| Harness 误拦正常修复 | 用户挫败 | 允许用户 override；持续调 patch 检测阈值 |
| Prompt 精进打断节奏 | 用户烦 | 简单 prompt 快速通道 + 可跳过 |
| 独立 IDE 工程量大 | 延期 | Monaco + Tauri；三大可视化分迭代交付 |
| AI 执行不可靠 | 核心体验崩 | 限定 TS/Next.js；git 自动回滚 |
| 与 Cursor 正面竞争 | 获客难 | 差异化：三大可视化 + Harness + 精进反馈 |
